Los estudiantes podrán: comprender la transición del contrato API al código real, e implementar rutas de servidor vinculadas a controladores usando JDoodle y GitHub para realizar pruebas de endpoints.

🤖 Prompt de Referencia para XOLO (Solo para Maestro)

⚠️ NOTA: XOLO ya conoce el tema de esta misión. Este prompt es SOLO de referencia en caso de emergencia técnica: 
Explícame qué es routing en un servidor y por qué una "ruta" funciona como una dirección que el backend escucha. Analicemos cómo traducir un endpoint del Documento de Diseño de API (método + ruta) a código real, y qué significa "conectar" la ruta con una función de controlador vacía. 🚨 Resolución Rápida de Problemas

  • Si los alumnos no pueden ingresar a Gnius Space: Activa XOLO SOS en xolo.gnius.club para continuar la misión sin interrumpir la clase.
  • Si el servidor no responde: Verificar que el puerto esté correctamente configurado y que el código no tenga errores de sintaxis. Revisar los logs de la consola.
  • Si GitHub da error al subir: Revisar que los cambios estén confirmados (committed) y que la conexión con el repositorio remoto sea correcta.
  • Si las pruebas de endpoints fallan: Verificar que el método HTTP (GET, POST, etc.) y la ruta coincidan exactamente con lo definido en el código. Usar Postman o Thunder Client para hacer pruebas más detalladas.
  • Si hay errores en el código: Recordar a los estudiantes que XOLO puede ayudar a depurar. Pídeles que copien el error y se lo muestren a XOLO.

📊 Diapositiva 3: Brain Spark

1. 💬 Pide a XOLO que te ayude a crear tus rutas:

"Ayúdame a implementar el esqueleto de rutas: elige conmigo 2–3 endpoints de mi diseño M05 (especificando método HTTP + ruta) y conviértelos en rutas reales en el servidor, vinculadas a controladores vacíos. Guíame a decidir una convención de nombres y a probarlas con respuestas temporales. Dame el código listo para agregar a mi servidor."

2. 🔬 Experimenta y descubre con tus rutas:

  • ¿El servidor responde con el mensaje esperado al acceder a la ruta?
  • ¿Qué sucede si cambias el método de la petición (ej. de GET a POST)?
  • ¿Funciona igual la ruta si usas mayúsculas o minúsculas?
  • ¿Cómo asegura el contrato API que el backend y el frontend hablen el mismo idioma?
  • ¿Por qué es importante la organización de código para decidir qué rutas son necesarias y cuáles no?

3. 📝 Prueba y documenta:

  • Usa Postman o Thunder Client para probar cada ruta
  • Captura las respuestas exitosas del servidor
  • Sube tu código actualizado a GitHub

¡Es hora de crear! Vamos a convertir sus planos en una estructura real. Usen el prompt en pantalla para que XOLO los guíe en la creación del esqueleto de sus rutas. El flujo es: revisen su diseño de API de la Misión 05, seleccionen 2-3 endpoints para implementar, trabajen con XOLO para generar el código, agréguen las rutas a su servidor, y pruébenlas. Por ejemplo, podrían crear una ruta GET /api/users para obtener usuarios y otra POST /api/posts para crear publicaciones. Usen Postman o Thunder Client para probar que responden correctamente. No olviden subir sus avances a GitHub y capturar las respuestas exitosas. Si tienen errores, pregúntenle a XOLO.
